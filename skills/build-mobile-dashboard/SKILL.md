> Версия 1.0 · Дата создания: 2026-05-14 · Автор: Иван И.

Этот скил **только для Claude в чате claude.ai**. Claude Code (CC) этот файл не выполняет — CC только верстает HTML по готовому JSON.

---

## ЦЕЛЬ

Из набора файлов, выгружаемых Иваном из ITIGRIS / банка, собрать **один JSON-файл** строго в формате `docs/JSON_SCHEMA.md` и положить его в `public/data/YYYY-MM-DD.json` + продублировать в `public/data/latest.json`.

**ВАЖНО: Claude не делает «улучшений», не округляет цифры, не придумывает данных.** Если поля нет в источниках — ставит `null` или 0 с пометкой в `meta.warnings[]`.

## АЛГОРИТМ ПОСТРОЕНИЯ JSON

### 1. `meta` (заголовок)
```
meta.date          = дата дня в формате YYYY-MM-DD
meta.date_display  = "DD.MM.YYYY, дн" (например "14.05.2026, чт")
meta.updated_at    = ISO-таймштамп с зоной +07:00 (Новокузнецк)
meta.version       = "1.0"
meta.source_files  = [список имён файлов, полученных от Ивана]
meta.warnings      = [строки с предупреждениями если что-то не сошлось]
```

### 2. `money_now` — остатки счетов

**Источник:** `ДДС_Май_*.xlsx` → лист «6. Дашборд» → строки 9–18.

**Карта строк** (по структуре файла Ивана):
| Строка | Счёт | JSON-имя |
|---|---|---|
| 9 | Р/с Сбер ООО | `"Р/с Сбер ООО"` |
| 11 | Касса Бардина | `"Касса Бардина"` |
| 12 | Касса Дружбы | `"Касса Дружбы"` |
| 13 | Сейф | `"Сейф"` |
| 14 | Одуван | `"Одуван (хоз)"` |
| 15 | Заначка ИН | `"Заначка ИН"` |
| 16 | Депозит ИН | `"Депозит ИН"` |

**ВАЖНО:**
- **Р/с КББ ИП (строка 10) — НЕ включать** в `money_now`. Он идёт в `overdraft.kbb_balance`.
- **ФОТ** — если есть в файле (строка может появиться), добавлять. Если 0 — не показывать.
- `money_now.total` = сумма всех включённых счетов. Сравнить со значением «ИТОГО без КББ ИП» в файле (строка 18) — должно совпасть.

**Сортировка для UI:** по убыванию значения.

**Статус:**
```
total > 1 000 000 → status_color="green",  status_label="Ликвидность OK"
total > 500 000   → status_color="amber",  status_label="Ликвидность средняя"
total < 500 000   → status_color="red",    status_label="Ликвидность низкая"
```

### 3. `overdraft` — параметры КББ

**Источник:** `КББ.png` (визуально парсить значения).

| Поле скрина | JSON |
|---|---|
| «Средств на счёте» | `kbb_balance` |
| «Сумма задолженности» | `used` |
| «Неиспользованный лимит кредита» | `free` |
| «Лимит кредита» | `limit` (всегда 1 500 000) |

**Расчёт:**
- `percent` = round(used / limit × 100)
- `color`: `< 60` зелёный, `60–85` амбер, `> 85` красный
- `due_date` = «15.06.2026» (фиксировано до уточнения)
- `monthly_interest` = 35 000 (фиксировано)

**Согласованность:** `used + free` должно быть = `limit`. Если нет — добавь warning.

### 4. `stock` — товарка

**`uncompleted_orders`:** из `uncompletedOrdersReport.xls`, строка «Итого: N».
| Колонка файла | JSON |
|---|---|
| col 1 «Итого: N» | `orders_qty` |
| col 8 «К доплате» | `to_pay` |

`older_30d` — пройти по строкам, посчитать у скольких заказов `сегодня − дата создания > 30 дней`.

**`warehouse_cost`, `warehouse_sale`, `suppliers_debt`:**
- если Иван не прислал свежий срез — взять значения из предыдущего JSON (`previous latest.json`)
- в `meta.warnings` написать `"stock: значения склада из <дата предыдущего среза>"`.

### 5. `payment_calendar`

**Источник:** `ДДС_Май_*.xlsx` лист «4. ДДС день» или отдельный лист «Платёжный календарь».

Алгоритм:
1. Собрать все плановые платежи на ближайшие 14 дней от `meta.date`.
2. Вычесть факт списаний (тот же лист, статус «Факт»).
3. `next_7_days` = сумма плановых платежей с `date+1` по `date+7`.
4. `next_14_days` = сумма плановых платежей с `date+1` по `date+14`.

**Если файл не содержит чёткого календаря:** взять стандартные ежемесячные платежи Ивана (см. справочник в `docs/PAYMENT_CALENDAR_DEFAULTS.md`) и вычесть фактические списания из ДДС.

### 6. `pending_payments` — из Faktura

**Источник:** `payments.html` (печатная форма НПП).

Алгоритм парсинга:
1. Разрезать файл по маркеру `"Плательщик"` — каждый кусок = одно поручение.
2. Для каждого извлечь:
   - сумма: `Плательщик Сумма XXX-YY` → `RUB.KOP`
   - получатель: блок до `Получатель`
   - назначение: блок до `Назначение платежа` (отрезать первое предложение, до 200 симв.)
   - дата: `Платежное поручение N <номер> ДД.ММ.ГГГГ Дата`
3. Разделить:
   - **future:** дата > meta.date → точно ещё не списан
   - **past_or_undated:** дата ≤ meta.date или нет даты → могли быть исполнены, нужна сверка
4. Сортировать каждую группу по убыванию суммы.

### 7. `loans`

Источник: предыдущий JSON + при необходимости свежий реестр кредитов от Ивана.

**Ключевые поля:**
- `priority: true` → подкраска красным (досрочка) — сейчас только KBB_OVER
- `warning: true` → подкраска жёлтым (% only) — сейчас только SPIRIDONOV
- `frozen: true` → приглушённый стиль — сейчас только PAVLIK

Эти флаги Иван меняет вручную, попросив Claude отметить тот или иной займ.

**Расчёт итогов** (для блока сверху):
- `total` = сумма всех `balance` где НЕ `frozen`
- `monthly_payment` = сумма всех `payment`
- `interest_part` = ~57% от `monthly_payment` (по факту апреля)
- `body_part` = `monthly_payment − interest_part`

### 8. `sales` — продажи дня

**Источник:** `salesProfitReport_Бардина.xls` + `salesProfitReport_Дружбы.xls`.

**`kpi.gross_profit`** = сумма колонки «Прибыль» по всем строкам обеих точек (строка «Итого»).
**`kpi.revenue`** = сумма колонки «Цена продажи».
**`kpi.items_qty`** = сумма колонки «Количество».
**`kpi.margin_percent`** = round(gross_profit / revenue × 100, 1)
**`kpi.cash`** = касса дня из `moneyMovementDetailsReport.xls` итог по всем счетам (только «Выручка»).
**`kpi.checks_qty`** = количество строк-чеков в `moneyMovementDetailsReport.xls`.
**`kpi.refunds`** = из `moneyMovementReport.xls` строки «Возврат денежных средств» + «Возврат денежных средств за услугу».

#### `glasses` (категория «Линзы» + «Оправы»)
- `lenses.qty/sum` ← Бардина «Линзы» + Дружбы «Линзы»
- `frames.qty/sum` ← Бардина «Оправы» + Дружбы «Оправы»
- `bardina` ← сумма обеих категорий Бардины
- `druzhby` ← сумма обеих категорий Дружбы
- `total_qty/total_sum` = bardina + druzhby

#### `mkl` (категория «Контактные линзы»)
По точкам.

#### `services` (категория «Услуги»)
**Группировка** (по полю «Параметры» в файле):
- содержит `"ОКТ"` → группа «ОКТ (все виды)»
- содержит `"УЗИ"` → «УЗИ глаза»
- содержит `"проверка зрения"` → «Бесплатные»
- содержит `"подбор МКЛ"` → «Бесплатные»
- содержит `"подбор очков"` → «Бесплатные»
- содержит `"консультация"` → «Консультация офт.»
- содержит `"ЛДВК"` или `"лазер"` или `"ПРЛКС"` → «ЛДВК (лазер) 1 глаз»
- содержит `"изготовление"` → «Изготовление очков»
- содержит `"периметр"` → «Периметрия»
- содержит `"Маклак"` → «Маклаков (ВГД)»
- содержит `"КЧСМ"` → «КЧСМ»
- содержит `"авторефр"` → «Авторефрактокератометрия»
- содержит `"диспансер"` → «Диспансерный осмотр»
- остальное → «Прочие»

Внутри группы суммировать qty и sum по обеим точкам.

#### `other`
Категории «Аксессуары», «Массовка», «Солнцезащитные очки» — суммарно по обеим точкам, отдельными строками.

### 9. `stoplight` (светофор недели в шапке)

Логика приоритета (срабатывает первое):
```
если overdraft.percent > 85 → red    + "Овердрафт критично"
если payment_calendar.next_7_days > money_now.total → red + "Разрыв на неделе"
если overdraft.percent > 70 → amber  + "Овердрафт NN%"
если money_now.total < 500 000 → amber + "Ликвидность низкая"
иначе → green + "Норма"
```

## ВАЛИДАЦИЯ ПЕРЕД ВЫДАЧЕЙ JSON

Перед тем как отдать Ивану итоговый файл, **Claude проверяет**:

- [ ] `money_now.total` = сумма всех `accounts[].value`
- [ ] `overdraft.used + overdraft.free = overdraft.limit`
- [ ] `pending_payments.total = future.total + past_or_undated.total`
- [ ] `pending_payments.future.total` = сумма `future.items[].amount`
- [ ] `pending_payments.past_or_undated.total` = сумма `past_or_undated.items[].amount`
- [ ] `loans.total` = сумма всех `loans.items[].balance` где НЕ `frozen`
- [ ] `loans.monthly_payment` = сумма всех `loans.items[].payment`
- [ ] `sales.glasses.total_qty` = `bardina.qty + druzhby.qty` = `lenses.qty + frames.qty`
- [ ] `sales.mkl.total_qty` = `bardina.qty + druzhby.qty`
- [ ] `sales.services.total_qty` = сумма `services.items[].qty`
- [ ] `sales.other.total_qty` = сумма `other.items[].qty`

Если что-то не сошлось — добавить строку в `meta.warnings[]` с описанием расхождения, **но JSON всё равно отдать**. Иван разберётся.

## ANTI-PATTERNS (НЕ ДЕЛАТЬ)

1. **Не вычислять цифры «по памяти».** Если данных нет — `null` + warning.
2. **Не округлять.** Все суммы — целые рубли (как в источниках). Маржа — 1 знак после запятой.
3. **Не менять структуру JSON.** Если нужно новое поле — сначала обновить `docs/JSON_SCHEMA.md`, потом скил.
4. **Не делать дашборд красивее.** За UI отвечает `public/dashboard.html`. Скил только формирует данные.
5. **Не парсить КББ-овердрафт из ДДС-файла.** Только из скрина банка.
6. **Не путать «касса дня» и «выручка по выдачам».** Касса (mmDetails) ≤ выручка (salesProfit) обычно, потому что выдачи идут сегодня, а оплачены раньше.
7. **«Невыданные заказы» = «К доплате»**, не «Оплачено» и не «Сумма со скидкой». Это деньги, которые **должны клиенты прийти и доплатить**.
